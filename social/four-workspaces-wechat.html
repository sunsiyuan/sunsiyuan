分享一下我的工作区怎么配的。为了并行多跑几个 agent，同一个项目在硬盘上拷了四份。

首先是电脑，两个主机，一台 mac mini，一台 MacBook pro。

中央是主工作区（注意力集中区，需要频繁交互思考），一般左右分屏两个 Terminal。

左屏幕是副屏，往往上下分屏，搭配比较自由——ChatGPT 上、terminal 下，浏览器上、terminal 下。一般放自己会跑长程的活；有些需要审阅的文档，会直接变成 html 或者 artifacts 从浏览器打开。

右屏幕是 MacBook pro，主要是微信、轻度的文档工作、X、信息获取。

注意力切分决定了注意力上限。

接下来是重度迭代一个项目时候的配置。

shengcai 是我 5 月主力推进的项目。推进过程中来来回回折腾了好几次工作区配置。一开始在一个主文件夹 ops-agent 里迭代，随着项目逐渐庞大，数据库的关联性越来越多，发现 agent 工作时长越来越长了，单个任务从一开始的 20 分钟慢慢变成 1 个半小时甚至3个小时（大规划分步执行workflow）。这中间不会想着等，而是想着其他可切分的、规划中的 feature 怎么落地。但是又不能直接 git checkout，所以一开始尝试了几次分 gittree，后来找到了一个妙招：把一个文件夹复制好几份出来。

为什么这个方法比 gittree 好？主要还是 checkout 不用瞎操心。

尤其是每个都需要单独的 Chrome CDP port、mock 数据去 verify 最终实现、后端端口。这时候四个文件夹非常方便——端口按照顺序 offset 就行。所以文件夹变成了 shengcai/ops-agent、shengcai/ops-agent-dev-1、shengcai/ops-agent-dev-2、shengcai/ops-agent-dev-3。

除了这几份，还有一个 build 文件夹，也是把整个文件夹原模原样复制了一遍，里面是个 windows build。

依赖完全不一样，属于动都不敢动，不然重新下载搞老半天。这也是一开始踩过的坑：客户说有苹果电脑，结果软件发过去装了一天都没装上，投屏一看，连 Applications 都不知道在哪，算了还是做 windows build 吧。

不过这个复制文件夹的方法后来又往回融了一点：磁盘不够用，就从父文件夹 shengcai/ 那层，把各份的 node modules 都软链到同一份，省空间。本来以为这跟 worktree 是一回事，其实不是——worktree 是共用一份 git 历史、工作文件各存各的；我这个反过来，各存各的 git、共用 node modules。

给一个概念，这是个多大的项目：从 4 月下旬开工，到客户稳定使用两个半月时间，主仓库 47 万行代码，4800 多个 commit、1000 多个合并的 PR、200 多个 issue。我贡献 70%，另一个同事 30%。我们两个没有写过一行代码。

说出来是有点土，比我讲究的做法肯定有。今天只是抛砖引玉，也想听听大家的实践。
